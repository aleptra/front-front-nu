This guide explains how to use {key} references inside JSON variable files to avoid repeating values like base URLs across multiple entries.

Overview

Variable files loaded via the var attribute support internal interpolation. Top-level string values can be referenced anywhere in the same file using the {key} syntax. References are resolved in a single pass at load time, before the data is cached.

Basic Usage

Define shared values as top-level keys, then reference them with {key} in nested values:

JSON (assets/json/vars/api.json) { "origin": "https://api.example.com", "auth": { "login": "{origin}/auth/login", "logout": "{origin}/auth/logout" } }

After loading, {origin} is replaced with the top-level value:

Resolved Result { "origin": "https://api.example.com", "auth": { "login": "https://api.example.com/auth/login", "logout": "https://api.example.com/auth/logout" } }

Advanced Example

Multiple top-level keys can be used across the file. Placeholders without a matching top-level key are preserved for runtime binding:

JSON (assets/json/vars/config.json) { "api": "https://api.example.com/v2", "cdn": "https://cdn.example.com", "endpoints": { "products": "{api}/products", "search": "{api}/search?q=", "profile": "{api}/users/{id}" }, "images": { "avatar": "{cdn}/img/avatars/", "banner": "{cdn}/img/banners/" } }
Note: {id} is not resolved because there is no top-level key called "id". It remains available for runtime binding via bindvar or bindasset.

Rules

  • Only top-level string values are used as sources.
  • Nested objects or arrays cannot be referenced.
  • Resolved in a single pass - references cannot chain.
  • Unresolved placeholders are preserved for runtime binding.
  • The pattern matches word characters: [a-zA-Z0-9_].

Best Practices

  • Define shared base values (origins, prefixes) as top-level strings.
  • Use short key names to keep URLs readable.
  • Switching environments only requires changing the top-level values.
